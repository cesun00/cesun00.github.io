On the Build System of Linux Kernel

Published at: September 6, 2023 | Last modification: July 25, 2024

Linux Makefile

This article analyzes the building process of the Linux kernel, including the implementation of Makefiles. Various build-time generated files and build techniques are discussed.

It is well-known that the Linux kernel should be built by make *config followed by a make call without a goal. The make *config command does nothing more than invoke a wizard program, which may include compiling that wizard program first. It’s the wizard program’s responsibility to properly generate a .config file which holds the user’s choice on the kernel building configs.

The wizard program usually reads various Kconfig files in the source tree to determine what kernel build options are devised by the kernel developer, and in what structure these options should be displayed to an end user. The Kconfig system won’t be discussed as a part of a kernel’s build here. See this article for an intro to the syntax of a Kconfig file and the rules of writing one.

Before a build, it’s recommended to invoke 1 of the 3 clean targets:

  • clean: Remove most generated files but keep the config and enough build support to build external modules
  • mrproper (more proper): Remove all generated files + config + various backup files
  • distclean (clean for distribution): mrproper + remove editor backup and patch files

out-of-source build

Linux supports out-of-source build by an O=dir make CLI argument:

1cd linux-6.8.2
2mkdir build
3make O=build defconfig
4cd build
5make -j12

In the Makefile, the source directory is referred to by the $(srctree) variable while the build directory is the $(objtree). If the O=dir argument is not used, both will be the same.

A kernel build is initiated by calling a plain make on the build directory. This Makefile-based system also must support future repetitive invocation of make by detecting modifications and building only the necessary parts. The make interface is also versatile and assumes jobs of different nature (clean, generate configs, build modules, build the vmlinux), bringing complexity to the implementation of Makefiles.

We will be exploring how the Makefile pieces are put together to achieve this.

Overview

The Makefile-based build system of Linux consists of 4 types of file:

  1. The root Makefile. It is the entrance of make execution.

  2. scripts/Kbuild.include and scripts/Makefile.*

    These are auxillary Makefile snippets that implement utilities and variables used by othe Makefiles, instead of defining build rules for the kernel.

  3. The Makefiles in each directory of source tree, including scripts/Makefile.

    These files append string values to certain well-known variables, including obj-y, obj-m, etc., rather than introducing new Makefile rules.

  4. Build time generated .<output_filename>.cmd and .<output_filename>.d in build tree, both being Makefile snipppets.

    They contain only Makefile rules (i.e. dependency) for <output_filename>.

scripts/Kbuild.include is included by many other Makefiles to provide utility functions. This section discusses the most commonly used ones.

cmd: print and execute commands

 1# $(call cmd, foo):
 2#
 3# with echo supressed:
 4# if `cmd_foo` exist, 
 5#       print either `quiet_cmd_foo` or `cmd_foo` value as string
 6#       trap interrupt
 7#       call cmd_foo
 8# else
 9#      no-op
10cmd = @$(if $(cmd_$(1)),set -e; $($(quiet)log_print) $(delete-on-interrupt) $(cmd_$(1)),:)

The cmd function should be used when you want to call certain Makefile functions and print to the terminal a message. It will respect the make V=[1|2] argument and determine whether to print verbose messages. You see prettified one-line messages during a kernel build thanks to this function, e.g.:

kbuild-file: get the path to the Makefile or Kbuild file under $(src)

1###
2# The path to Kbuild or Makefile. Kbuild has precedence over Makefile.
3kbuild-dir = $(if $(filter /%,$(src)),$(src),$(srctree)/$(src))
4kbuild-file = $(or $(wildcard $(kbuild-dir)/Kbuild),$(kbuild-dir)/Makefile)

The caller must define an src variable before calling kbuild-file, which must be a directory name without the trailing slash.

For example:

1# snippet from: scripts/Makefile.build
2
3# obj is an export-ed variable to this submake
4src := $(obj)
5
6# later ...
7
8# include `$(obj)/Kbuild` or `$(obj)/Makefile`
9include $(kbuild-file)
  • kbuild-file expands to the absolute path to the Kbuild file, or if absent, the Makefile, under src.
  • kbuild-dir is a helper macro never used independently.

Use these macros, instead of hardcoding $(src)/Makefile, for compatibility. Historically Makefile pieces shattered in each directory have the file name Kbuild. Almost all of them have been renamed to Makefile by the time of this writing, but some old device drivers may be still using Kbuild.

$(build): standard way of invoking sub-make

$(Q)$(MAKE) $(build)=<dir> is the recommended way to invoke make in a subdirectory. ($(Q) expands to @ or empty, determining whether to echo the recipe.)

You will see recipes like the following one pervasively in Makefiles:

1%config: outputmakefile scripts_basic FORCE
2	$(Q)$(MAKE) $(build)=scripts/kconfig $@

build is defined as:

1###
2# Shorthand for $(Q)$(MAKE) -f scripts/Makefile.build obj=
3# Usage:
4# $(Q)$(MAKE) $(build)=dir
5build := -f $(srctree)/scripts/Makefile.build obj

That is, without changing the current working directory, $(MAKE) $(build)=dir some-target uses scripts/Makefile.build as the Makefile, and executes its some-target goal with an obj=dir argument, where dir must be a directory name without the trailing slash. Most often, executing the default goal is desired, so some-target is omitted.

scripts/Makefile.build: the module builder

The purpose of scripts/Makefile.build is to make life easier for kernel module maintainers, such that only variable definition, and no rule, needs to be defined in a module-specific Makefile (rules can be added for TODO, see below).

When building a kernel subdirectory, the recommended $(MAKE) $(build)=dir syntax always runs a sub-make with the Makefile scripts/Makefile.build (via the -f option, see build for detail).

Among other scripting, this file includes the $(kbuild-file) (i.e. the Makefile in the dir as given in $(build)=dir, see kbuild-file for detail), and load a scripts/Makefile.lib to normalize the variables defined in the $(kbuild-file):

 1# scripts/Makefile.build
 2
 3# init variables, load util functions, load compiler setups ...
 4include $(kbuild-file)
 5include $(srctree)/scripts/Makefile.lib
 6
 7# Do not include hostprogs rules unless needed.
 8# $(sort ...) is used here to remove duplicated words and excessive spaces.
 9hostprogs := $(sort $(hostprogs))
10ifneq ($(hostprogs),)
11include $(srctree)/scripts/Makefile.host
12endif

techniques: Ensure a file has been generated

file-name target with no dependency is considered always up-to-date, unless that file doesn’t exist (i.e. effectively becomes a PHONY):

 1# $(KCONFIG_CONFIG) defaults to `.config`.
 2# Whenever make finds that the `.config` file doesn't exist thus the recipe needs to be executed, it's a problem
 3
 4$(KCONFIG_CONFIG):
 5	@echo >&2 '***'
 6	@echo >&2 '*** Configuration file "$@" not found!'
 7	@echo >&2 '***'
 8	@echo >&2 '*** Please run some configurator (e.g. "make oldconfig" or'
 9	@echo >&2 '*** "make menuconfig" or "make xconfig").'
10	@echo >&2 '***'
11	@/bin/false
