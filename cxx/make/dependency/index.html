Makefile as a Dependency Description Language

Published at: August 9, 2023 | Last modification: April 17, 2024

GNU Linux Make Makefile
  • https://www.gnu.org/software/make/manual/html_node/Double_002dColon.html
  • https://www.gnu.org/software/make/manual/html_node/Remaking-Makefiles.html

The Makefile language can be divided into 2 parts, a dependency description language, and a scripting language including constructs like functions, variables, loops, branching etc.

This article discusses everything about the Dependency Description ability of a Makefile. See TODO for the scripting part.

(Explicit) Rules

The unit of dependency description is explicit rules (often, just rules). A rule has the following syntax:

1# each structure below is named a "rule"
2target: prerequisites
3    recipe
4
5# e.g.
6foo.o: foo.c
7        $(CC) -Iinclude/ -Iarch/x86/include foo.c

An explicit rule describes when and how to remake one or more files, called the rule’s targets. It lists the other files that the targets depend on, called the prerequisites of the target, and may also give a recipe to use to create or update the targets.

the File nature of target

An important assumption in the design of make and Makefile is that: target names are file names.

  • This is not always true, as make all is invoked from the command line

Targets that are meant to be invoked directly on the command line by the end users are called goals. When make is invoked without a goal, a default goal will be invoked: it is the value of special variable .DEFAULT_GOAL, or if omitted, the first target whose name doesn’t start with ..

⚠️

Rules are called explicit if they are present in a Makefile, as opposed to implicit rules (or sometimes built-in rules).

GNU Make attempts to use the following rule when make foo.o is requested (or used as a dependency) and a foo.c file exists, but no rule of making foo.o presents in the Makefile:

1foo.o:
2	$(CC) -c foo.c

Similarly, a few other implicit rules are hardcoded for other suffixes, e.g. a bar.c file from a bar.l (lex source) file. See the manual page for an exhaustive list.

Implicit rules are active even when no Makefile exists. Type make main.o in a directory with main.c and no Makefile and see what happens.

Users can change the value of the variable CC, thus changing the compiler used by such implicit rules. Such variables are known as built-in variables.

Implicit rules and builtin variables won’t be discussed in this series. They are legacy features, whose functionality can be completely superseded by using patterned rules. It is strongly suggested to disable them at the beginning of a Makefile by (code from Linux Kernel):

(click to expand)
 1ifneq ($(sub_make_done),1)TODO
 2
 3# Do not use make's built-in rules and variables
 4# (this increases performance and avoids hard-to-debug behaviour)
 5MAKEFLAGS += -rR
 6
 7# if you are using GNU Make 4.0 (released in Oct 2013), ignore the following
 8
 9ifneq ($(filter 3.%,$(MAKE_VERSION)),)
10# 'MAKEFLAGS += -rR' does not immediately become effective for GNU Make 3.x
11# We need to invoke sub-make to avoid implicit rules in the top Makefile.
12
13need-sub-make := 1
14endif
15
16
17ifeq ($(need-sub-make),1)
18
19PHONY += $(MAKECMDGOALS) __sub-make
20
21$(filter-out $(this-makefile), $(MAKECMDGOALS)) __all: __sub-make
22	@:
23
24# Invoke a second make in the output directory, passing relevant variables
25__sub-make:
26	$(Q)$(MAKE) $(no-print-directory) -C $(abs_objtree) \
27	-f $(abs_srctree)/Makefile $(MAKECMDGOALS)
28
29else # need-sub-make
30
31# all rules goes here, till the end of the Makefile
32
33endif # need-sub-make

Patterned Rules

% matches 1 or more characters. You need 2 rules to achieve the same effect as the * wildcard in glob:

1config: outputmakefile scripts_basic FORCE
2	$(Q)$(MAKE) $(build)=scripts/kconfig $@
3
4%config: outputmakefile scripts_basic FORCE
5	$(Q)$(MAKE) $(build)=scripts/kconfig $@

Slash-separated segments are not treated specially:

1# matches both foo/config/auto.conf and bar/zoo/config/auto.conf
2%/config/auto.conf:
3	@:

Can appear multiple times:

1$(obj)/%.asn1.c $(obj)/%.asn1.h: $(src)/%.asn1 $(objtree)/scripts/asn1_compiler
2	$(call cmd,asn1_compiler)

Misc features

Multiple Occurrences of the same Target

A target name (usually a file name) can appear in multiple rules, and only 1 of them can have a recipe:

 1__all:
 2
 3# later... 
 4
 5ifeq ($(need-sub-make),1)
 6
 7$(filter-out $(this-makefile), $(MAKECMDGOALS)) __all: __sub-make
 8	@:
 9
10else
11# omitted
12endif
13
14# later... 
15
16ifeq ($(KBUILD_EXTMOD),)
17__all: all
18else
19__all: modules
20endif

This feature allows dependencies to be added conditionally and incrementally.

techniques

Phony as dependency is function calls

1.PHONY: main func_foo
2
3main: func_foo
4	@echo "but there is no argument passing anyway..."
5
6func_foo:
7	@echo "do you know that having phony target as prerequisites feels like function calls?"

another example from kernel:

1$(MAKECMDGOALS): __build_one_by_one
2	@:
3
4__build_one_by_one:
5	$(Q)set -e; \
6	for i in $(MAKECMDGOALS); do \
7		$(MAKE) -f $(srctree)/Makefile $$i; \
8	done
