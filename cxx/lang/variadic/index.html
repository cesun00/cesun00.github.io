Variadic Arguments in C/C++

Published at: September 7, 2023 | Last modification: April 29, 2024

C++ Programming Language

Variadic Macro Functions

Preprocessors of gcc and clang supports variadic macro function. This feature is mandatory since C99 / C++11.

 1// An ellipsis as the last macro parameter becomes the sink.
 2// Use `__VA_ARGS__` to refer to whatever token sequence the sink accepts.
 3#define foo(...) real_foo(42, __VA_ARGS__)
 4
 5// GNUCPP-specific named sink:
 6#define foo(bar...) real_foo(42, bar)
 7
 8// Preceding normal arguments are allowed, with some problems
 9#define eprintf(format, ...) std::fprintf(stderr, format, __VA_ARGS__)
10
11eprintf("hooray!");     // expansion error in all-time C and prior to C++20.
12eprintf("hooray!", );   // Expands to   std::printf(format, );     and compiler will complain.
13
14
15// C++20
16// 1) implicitly assumes the argument comma if omitted; i.e. eprintf("hooray!") is equivalent to eprintf("hooray!", )
17// 2) introduces `__VA_OPT__(X)` which expands to X if __VA_ARGS__ is not empty, or nothing otherwise.
18#define eprintf(format, ...) std::fprintf(stderr, format __VA_OPT__(,) __VA_ARGS__)
19
20eprintf("hooray!");             // expands to   std::fprintf(stderr, "hooray");
21eprintf("%d hooray!", 42);      // expands to   std::fprintf(stderr, "%d hooray", 42);

Function with Variadic Arguments

in C

Variadic arguments in C is introduced by an ellipsis ... in the function declaration which must appear last in the parameter list and must follow at least one named parameter. The ... and the proceeding parameter must be delimited by a comma:

1int printx(const char* format, ...);

As the implementor of printx, do:

 1#include <cstdarg>
 2
 3int printx(const char* fmt...) {
 4    // 1. Allocate an instance of va_list on stack.
 5    va_list args;
 6    // 2. Initialization.
 7    va_start(args, fmt);
 8
 9    // 3. call va_arg with type
10    int a = va_arg(args, int);
11    double b = va_arg(args, double);
12    const char *const c = va_arg(args, const char*);
13
14    // 4. destruction
15    va_end(args, fmt);
16}

There is no limitation as for what can be used as arguments: primitive types, pointers, and struct instance. If the callee va_args() failed to expect, dirty data is read, and your program booms.

in C++

The leading comma is omitted from C++’s standard syntax, but writting one is still allowed for C compatibility. C++ also allows variadic functions without leading named ordinary parameter, but portability of the compiled binary of such functions is not guaranteed.

1int printx(const char* fmt...);
2int printx(const char* fmt, ...); // same as above.
3int printx(...); // allowed in C++ but not in C

access & implementation

 1#include <cstdarg>
 2
 3int printx(const char* fmt...) {
 4    // 1. Allocate an instance of va_list on stack.
 5    va_list args;
 6    // 2. Initialization.
 7    va_start(args, fmt);
 8
 9    // 3. call va_arg with type
10    int a = va_arg(args, int);
11    double b = va_arg(args, double);
12    const char *const c = va_arg(args, const char*);
13
14    // 4. destruction
15    va_end(args, fmt);
16}

  1. va_start and va_args are macros. (mandated by spec?)

  2. It’s undefined behavior if va_arg is called when there are no more arguments in va_list.

  3. Spec leave content of va_list unspecified. It has to be stateful (i.e. hold a cursor) in order to implement such API pattern.

    sizeof(va_list) under amd64 gcc is 24 bytes.

    1typedef __builtin_va_list __gnuc_va_list;
2typedef __gnuc_va_list va_list;

Variadic parameters are essentially unnnamed and untyped. Caller push whatever arguments onto its own (?) stack frame, and make a callq instruction on x86.
