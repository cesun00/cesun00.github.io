Template Specialization

Published at: August 3, 2023 | Last modification: June 15, 2024

C++ Programming Language template

Specialization allows a complete different template definition to be used when specific 1) template arguments, or 2) patterns of template arguments is provided.

  1. is the case for both full and partial specialization
  2. is more expressive, and is only the case for partial specialization.

Specialization embodies ad-hoc polymorphism of metaprogramming: the same template name (interface) works for different type by really having multiple templates definition. It resembles function overloads, and is the the metaprogramming counterpart of the latter in many ways.

Full specialization

Full specialization allows a complete different template definition to be used when specific template arguments is provided.

All template types (function template / class template / variable template / etc.) can have full specialization.

Its syntax is landmarked by keyword template followed by an empty pair of <>, with exact template arguments upon which this specialization should be selected for, spelled after the unqualifed name of the primary template, in another pair of <>:

 1// function template - primary template
 2template <typename T, std::size_t N>
 3void foo() {
 4    std::printf("primary template - %s\n", __PRETTY_FUNCTION__);
 5}
 6
 7// function template - full specialization
 8template <>
 9void foo<const int, 42>() {
10    std::printf("full specialization - %s\n", __PRETTY_FUNCTION__);
11}
12
13int main() {
14    foo<int, 42>();       // primary template - void foo() [with T = int; long unsigned int N = 42]
15    foo<const int, 42>(); // full specialization - void foo() [with T = const int; long unsigned int N = 42]
16}

Partial Specialization (class template / variable template only)

Syntax:

1template<partial-specialization-paramter-list>
2class-key primary-template-name<template-argument-list> {
3    // ...
4}

where template-argument-list must contains at least 1 parameter from partial-specialization-paramter-list. i.e. template-argument-list can’t be already fully determined (that’s the syntax for full specialization).

 1// class template - primary template
 2template <typename T, std::size_t N>
 3struct MyArray {
 4    int x = 42;
 5};
 6
 7// class template - partial specialization
 8template <typename E>
 9struct MyArray<E, 42> {
10    int y = 55;
11};
12
13// class template - full specialization
14template <>
15struct MyArray<int, 42> {
16     int z = 121;
17};
18
19int main() {
20    MyArray<double, 0> a;       // select primary template
21    std::printf("%d\n", a.x);
22    std::printf("%d\n", a.y);   // unknown y
23    std::printf("%d\n", a.z);   // unknown z
24
25    MyArray<std::string, 42> b; // select partial specialization <E,42>
26    std::printf("%d\n", b.x);   // unknown x
27    std::printf("%d\n", b.y);
28    std::printf("%d\n", b.z);   // unknown z
29
30    MyArray<int, 42> c;         // select full specialization<int,42>
31    std::printf("%d\n", c.x);   // unknown x
32    std::printf("%d\n", c.y);   // unknown y
33    std::printf("%d\n", c.z);
34}

The sole purpose of partial-specialization-paramter-list is to introduce template parameters (as valid placeholders) in this specialization. It won’t change the template’s interface to clients, e.g. won’t allow you to pass less / more number of template argument upon instantiation whatsoever. Client must always respect the interface of primary template (i.e. number of / type-or-non-type-ness of template parameter), regardless of whether primary template or any specialization will be used.

pattern-based specialization resolution

Partial specialization are more expressive, in that it allows a specialization to be selected due to arguments satisfying certain traits, or exhibits certain patterns, rather than being a specific concrete type, compared to full specialization:

 1// primary template
 2template<typename T>
 3struct Foo {
 4    Foo() {std::puts("Foo primary template");}
 5};
 6
 7// partial specialization selected when T is a (possibly cv-qualified) pointer
 8template<typename E>
 9struct Foo<E*> {
10    Foo() {std::puts("Foo partial specialization for pointer");}
11}; 
12
13// primary template
14template<typename T, typename E>
15struct Bar { int x; };
16
17// partial specialization selected when T == E
18template<typename G>
19struct Bar<G,G> { int y; };
20
21int main() {
22    Foo<int> f0;            // Foo primary template
23    Foo<int*> f1;           // Foo partial specialization for pointer
24    Foo<const int*> f2;     // Foo partial specialization for pointer
25
26    Bar<int,double> a;          // hits primary Bar
27    std::printf("%d\n", a.x);
28    std::printf("%d\n", a.y);   // error: struct Bar<int, double>’ has no member named ‘y’
29
30    Bar<int, int> b;            // hits T = E specialization
31    std::printf("%d\n", b.x);   // error: ‘struct Bar<int, int>’ has no member named ‘x’
32    std::printf("%d\n", b.y);
33
34    Bar<double, double> c;      // hits T = E specializatoin
35    std::printf("%d\n", c.x);   // error: ‘struct Bar<double, double>’ has no member named ‘x’
36    std::printf("%d\n", c.y);
37}

Misc

Source order between specializations and instantiations

Specialization must be declared before the first (explicit or implicit) instantiation, in every translation unit.

 1template<class T> // primary template
 2void sort(std::vector<T>& v) { /*...*/ }
 3
 4int main() {
 5    std::vector<int> dt{1,2,3};
 6    sort(dt);   // implicitly instantiate sort<int> from primary template
 7}
 8
 9template<>
10void sort(std::vector<int>& v); // error: declaration of specialization found too late

Primary template and its specializations can be very different animals

Unlike inheritance, a specialization doesn’t get any piece of code from its primary template. The only thing they share is the same interface (name and template parameter list) to client.

If code sharing is intended, user must manually factor out the common code and use other mechanisms, (TODO: find an example from authoritative code)

The primary template is even not required to have definition in order to allow specialization. A declaration would suffice.

 1template <typename Type>
 2class Foo;
 3
 4template <>
 5class Foo<int> { };
 6
 7int main(int argc, char *argv[]) 
 8{
 9    Foo<int> f; // Fine, Foo<int> exists
10    Foo<char> fc; // Error, incomplete type
11    return 0;
12}

The extensional nature of template specialization

Specialization has an extensional nature: everyone can specialize a primary template as long as the definition of the primary template appears earlier in the same translation unit. This means you can specialize templates from included library header and even standard library header, and totally changes its definition:

 1// full specialization of std::vector
 2template<>
 3class std::vector<int> {
 4public:
 5    int bar;
 6};
 7
 8
 9// partial specialization of std::pair
10template<typename T>
11class std::pair<int, T> {
12public:
13    std::string_view value{"pair can be whatever I want"};
14};
15
16int main() {
17    using std::cout;
18    using std::endl;
19
20    std::vector<int> ff;
21    cout << ff.bar << endl;
22    cout << sizeof(ff) << endl;     // 4
23    // cout<< ff.size() << endl;    // error: no member named 'size' in 'std::vector<int>'
24    std::pair<int,double> my_pair;
25    cout << my_pair.value << endl;
26}

Such mechanics is frequently used in metaprogramming as compile-time user callback: library author simply use library_namespace::query<T> when coding library_namespace::great_algorithm<T> templates, and expects the user to provided a proper specialization of query on whatever same T they try to instantiate great_algorithm with. Templates like query exists sorely to let user specialize them, and the primary template can be as simple as no-op, or only works for a very limited range of types, or even incomplete (i.e. only declaration).

Good exmaple being class template std::hash<T> used by hash-based associative containers i.e. unordered_map and unordered_set. Objects of an instantiation of this template is expected to provide an operator() as non-static member function and return a std::size_t hash value. Specification leave the definition of its primary template unspecified, and only requires vendors to provide specialization for certain common types. Apparently different T requires different ways of computing hash, and primary template can’t do much on that. In gcc stdlibc++, the primary template only works for enumeration, and causes SFINAE if instantiated with non-enum T. If user want to use a custom type as key for unordered map or set, they must provide specialization of std::hash on that type.

side note: std::hash is not a function template, because function template can’t be partially specialized as far as C++23. std::hash must support partial specialization so that people can have things like specialization for all pointer type template<typename T> class std::hash<T*>. Making it a class template which provides an operator() is simply a workaround against that.

as a way of externally associating traits with type in metaprogramming

We hope to associate certain type definition with MyType:

 1// approach #1 (intrusive)
 2class MyType1 {
 3    public:
 4        using foo_type = int;
 5        // or arbitraryly complicated type manipulation.
 6        // Usually MyType itself is a business class template MyType<T>, and rhs of foo_type is parameterized on T
 7};
 8
 9// approach #2 (non intrusive)
10class MyType2 {
11
12};
13
14// Awkawrdly a class template because T-parameterized `using` or `typedef`
15// type reansoning "expression" can't exist on their own, thus must be class member.
16template<typename T>
17struct QueryFooType; // prevent instantiation of primary template
18
19template<>
20struct QueryFooType<MyType2> {
21    using foo_type = int;
22};
23
24// Since c++14 you can alias the QueryFooType<T>::foo_type via the following *type alias template*.
25//
26// std type traits library does exactly the same, provides class template for traits query with companion alias helpers.
27//
28// But still there is no way to get rid of the awkward usage of class template.
29// It would be nice if we have something like "type alias template specialization", which would eliminate the need for traits query class templates:
30//  
31//      // "primary type alias template"
32//      template<typename T>
33//      using query_foo_type_cool;
34//      
35//      // "type alias template full specialization"
36//      template<>
37//      using query_foo_type_cool<MyType2> = int;
38//      
39//      // "type alias template partial specialization"
40//      template<typename E>
41//      using query_foo_type_cool<GenericMyType<E>> = /* T-parameterized type reasoning */;
42// 
43// But so far, spec says no.
44//
45template<typename T>
46using query_foo_type_t = typename QueryFooType<T>::foo_type;
47//                          ^ keyword `typename` is mandatory (the dependent name rule)
48
49int main() {
50    MyType1::foo_type a;
51    // is equivalent to
52    query_foo_type_t<MyType2> b;
53    // is library's sugar to
54    QueryFooType<MyType2>::foo_type c;
55}
