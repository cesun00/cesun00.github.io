SLF4j

org.slf4j:slf4j-api provides APIs directly used by end user.

For each facaded logging framework, an adapter package, known as the “SLF4J binding”, is required to present on the classpath (and discovered later via SPI). If no one is found, slf4j defaults to a no-op implementation and emit a warning to stderr.

Do not place more than one binding on your (runtime) class path.

and here goes the chaos:

  • org.slf4j:slf4j-log4j12, maintained by slf4j, was the binding for log4j <= 1.2, which is now (202203) redirect to org.slf4j:slf4j-reload4j via maven’s <relocation> mechanism. Don’t use.

  • org.slf4j:slf4j-reload4j: binding for reload4j and log4j 1.2

  • org.apache.logging.log4j:log4j-slf4j-impl, maintained by log4j, should be used with log4j 2 and SLF4J <= 1.7.x.

    This artifact depends on (compile scope) org.slf4j:slf4j-api, org.apache.logging.log4j.log4j-api and (runtime scope) org.apache.logging.log4j.log4j-core.

    Thus this artifact, in extreme case, is the only artifact end user needs to explicity import in order to use slf4j + log4j combo. But specifying those dependeicies again will override their versions.

  • org.apache.logging.log4j:log4j-slf4j18-impl, maintained by log4j, should be used with log4j 2 and SLF4J >= 1.8.x

    ditto.

  • org.slf4j:slf4j-simple: an naive minimal implementation.

  • ch.qos.logback:logback-classic: logback binding to slf4j

    logback is develop by the same group of guys as slf4j. Thus this is claimed to be the “native” implementation.

Configuration

The idea is that application developer (who program only to slf4j-api), and end user (or deployer, e.g. tomcat admin), are different roles:

Embedded components such as libraries not only do not need to configure the underlying logging framework, they really should not do so. They should invoke SLF4J to log but should let the end-user configure the logging environment. When embedded components try to configure logging on their own, they often override the end-user’s wishes. At the end of the day, it is the end-user who has to read the logs and process them. She should be the person to decide how she wants her logging configured.

SLF4J Backends - log4j-1 and reload4j project

Initiated by Ceki Gülcü, the original author of Apache log4j 1.x, the reload4j project is a fork of Apache log4j version 1.2.17 with the goal of fixing pressing security issues. It is intended as a drop-in replacement for log4j version 1.2.17. By drop-in, we mean the replacement of log4j.jar with reload4j.jar in your build with no source code changes in .java files being necessary.
