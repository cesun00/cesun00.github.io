Java Service Provider Interface (SPI)

Published at: August 28, 2023 | Last modification: July 26, 2024

Java

This article introduces the Java Service Provider Interface (SPI). SPI workflow in Java 8 and before is first discussed, followed by the improvement brought by Java 9.

Motivation

A well-designed library should expose only interfaces to its clients, which are contracts that seldom change. Those codes invisible to the client - the implementation details - can thus be improved without breaking the contract.

  • From the library authors’ perspective, it is ideal that the client programmers program to interfaces or interfacial classes, to the extent that no implementation is even present during compile time. In practice, such a library is usually released as 2 different JARs. That is, javac’s classpath contains only the interfacial jar.
  • From the client’s perspective, library instances are returned from factory methods by interface type, which is the reason why you almost can’t see constructor calls when using such a library. The real challenge boils down to how such a factory method locates actual implementation instances of given interfaces at runtime.

There are good reasons for doing so:

  1. This prevents users from accessing internal APIs since downcasts to implementation won’t compile;
  2. When there are multiple implementation jars, the choice of implementation can be deferred to deployment time, perhaps made by a different person (DevOps Admin, instead of the application developer).
  3. This allows the separation of parties: API designers and implementation authors may come from different parties. The API jar is published before any implementation jar is released.

JDBC As an Example

In fact, the practice of separating API and implementation jars has long existed.

A good example is the JDBC driver, where a user programs to the java.sql.* APIs which are parts of standard JDK rt.jar; the vendor’s implementation is only required at runtime, e.g. for MySQL it’s mysql-connector-java-8.X.XX.jar. At compile time, client calls java.sql.DriverManager#getDriver to obtain a java.sql.Driver reference, which refers to a com.mysql.cj.jdbc.Driver instance at runtime.

Let’s see how the aforementioned challenge is solved traditionally: java.sql.DriverManager maintains a static registry, a list of known Driver implementations:

1private final static CopyOnWriteArrayList<DriverInfo> registeredDrivers = new CopyOnWriteArrayList<>();

Each implementation vendor is expected to report their Driver implementation by adding an implementation instance to this registry. This is always done by a static initializer. mysql-connector-java does the following:

 1package com.mysql.cj.jdbc;
 2
 3public class Driver extends NonRegisteringDriver implements java.sql.Driver {
 4    // Register ourselves with the DriverManager.
 5    static {
 6        try {
 7            java.sql.DriverManager.registerDriver(new Driver());
 8        } catch (SQLException E) {
 9            throw new RuntimeException("Can't register driver!");
10        }
11    }
12}

This workflow suffers from several problems:

  1. It’s the user’s responsibility to ensure the static initializer is executed, making necessary the infamous Class.forName() call in the past. Such a contract is not common practice.
  2. Using static initializers is considered a bad idea in most languages.
  3. An implicit contract is also imposed on the Driver implementation vendors: they have to read the specification and know what must be done in the static initializer.

SPI was developed to facilitate a workflow without these problems.

SPI

SPI (Service Provider Interface) is Java’s standard mechanism for discovering external implementations of a given interface at run time. Any caller of static method java.util.ServiceLoader#load(Class<S> service) obtains a ServiceLoader<S> instance. A ServiceLoader<S> instance represents the collection of all implementations of interface S presented on the run time classpath, provided that they are exposed by,

  1. in Java 8 and before, the implementation vendor JAR, in a META-INF/services/<interface_name> file
  2. in Java 9 and later, the provides directive in module-info.java.

For each discovered implementation, a ServiceLoader.Provider<S> instance is constructed, which is simply a tuple of Class<SImpl> and a lazy getter that returns an instance of that implementation. The ServiceLoader<S> instance holds the collection of such Provider<S>s and is ready to accept metadata queries or instantiation requests.

SPI is mainly used for 2 reasons:

  1. allows client programmers to program against interfaces whose implementation doesn’t exist at compile time, but will be discovered at run time; i.e. defers the choice of implementation to deployment time.
  2. gives a chance where implementation jars can receive callbacks to self-initialization code, and do not rely on class static initializer. e.g. Spring MVC uses this feature to implement its WebApplicationInitializer.

Java 8, And Before

To use SPI in Java 8 and before:

  1. for client user: one still obtains interface instances by calling factory methods from API jar;
  2. for API jar vendor: the factory methods discover implementations and obtain their instances at run time by calling static method java.util.ServiceLoader#load
  3. for implementation provider: implementation JAR exposes the implementation classes in a META-INF/services/<interface_name> file on the classpath.
    • the file name is the fully qualified name of the service interface / abstract class, e.g. java.sql.Driver; and
    • the file content is multiple lines of fully qualified name of the implementation this jar provides, one per line. e.g. com.mysql.cj.jdbc.Driver

All the static registration magics are now extracted to ServiceLoader: instead of asking on implementation vendor to write ad-hoc code that registers themselves, standard ServiceLoader now reads the well-known declaration file and load those classes followed by instantiating an instance.

It should be clear that SPI is a complete SDK-level solution, meaning that there is no magic from the compiler or JVM involved. ServiceLoader is simply a helper class for whoever doesn’t want to scan the classpath and read META-INF/services/* themselves.

Arbitrary operations can then be done with that instance, including:

  1. invoking implemented methods that run arbitrary code contained in the implementation. This is what Servlet 3.0 Specification does for javax.servlet.ServletContainerInitializer. Spring MVC receives a callback from Tomcat which in turn call implementations of its own interface org.springframework.web.WebApplicationInitializer from Spring user.
  2. exploiting the side-effect that at the time of instantiation, the class static initializers are run. This is what java.sql.DriverManager does to ensure the registration of Driver implementation.

Java 9, and later

It turned out that the community wasn’t very happy with the META-INF/service/* file-based solution.

With the advent of Java 9 module system, the new workflow was improved to exploit directives in the new module-info.java:

  1. for client user: nothing changes
  2. for API module jar vendor: declares that it will consume a service by the uses interface.name (or abstract class) directive
  3. for implementation module jar provider: declare that it will provide implementations of a service by the provides interface.name with implementation.name; directive.

Also, no change to the Java code on the API jar vendor’s side: the ServiceLoader pattern still has to be used.
