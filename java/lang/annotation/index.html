Annotation and Interface

Annotation and interfaces are twin constructs in Java that share a lot of common characteristics.

Annotations were designed to replace the now considered bad practice of marker interface. (Examples of marker interfaces are Serializable and Cloneable, which still exist weirdly in modern Java and break the design boundary between language SDK and JVM.) They mark certain properties of the annotated entity, be it a class, field, method, or method parameter.

Annotation is defined with the @interface syntax, just a @ away from an interface. At the language level, each interface @Foo {...} behaves almost the same as an interface Foo extends java.lang.annotation.Annotation {}:

  1. You can check the subinterface relationship with reflection:

    1System.out.println(BaseAnno.class.isInterface()); // true
2Arrays.stream(BaseAnno.class.getInterfaces()).forEach(System.out::println);  //interface java.lang.annotation.Annotation

  2. Since interface Annotation defines 4 methods (3 of which are implemented by Object), and Foo is effectively an empty subinterface, you can even have an implementation class of Foo.

    java.lang.annotation.Annotation looks like:

    1public interface Annotation {
2    // effectively the only method an implementation must provide
3    Class<? extends Annotation> annotationType();
4
5    // always implemented by all `Object`s, but listed here for distinct semantics requirement.
6    boolean equals(Object obj);
7    int hashCode();
8    String toString();
9}

    Now feels this black magic:

     1@interface FooTag {}
 2
 3// it's weird to have implementation of annotation, but legal
 4class FooTagImpl implements FooTag {
 5    @Override
 6    public Class<? extends Annotation> annotationType() { return null; }
 7}
 8
 9// later in code ...
10FooTag x = new FooTagImpl(); // ok

    It’s just fun to know this is possible I’m not aware of any use of this feature in the real world.

Yet, the similarity is an illusion. Interfaces and annotations are differentiated at the bytecode level, and the binary classfile format have several bit flags indicating that a .class file is an annotation instead of an interface. So what you’ve observed is more complicated than a javac syntax sugar.

More to note:

  1. manually writing an normal interface Bar extends Annotation {} does not make it an annotation.
  2. The fact that @Foo is a subinterface of Annotation somewhat explains why you can’t have extends or implements keyword after @interface Foo. But again, we are already studying black magics, so don’t take this too seriously.

Repeatable Annotation

Most annotations encountered from day to day are non-repeatable: It’s a syntax error if @Component or @Configuration is tagged twice on your Spring bean class.

Repeatable annotations are those which can be tagged more than once. To define a repeatable annotation, you must:

  1. define annotation MyRepeatable itself with @Repeatable(value = ...) tagged, where the value field must be the Class instance of its so-called container annotation companion.
  2. define the container annotation MyRepeatableContainer, whose value must be of type MyRepeatable[].
  3. ensure @Target.value() of MyRepeatable is a superset of that of MyRepeatableContainer.
  4. ensure MyRepeatableContainer has a longer @Retention.value than MyRepeatable.

Equivalence of Container and Multiple Occurrences

Given

1@Retention(RetentionPolicy.RUNTIME)
2public @interface FooContainer {
3    Foo[] value();
4}

And

1@Retention(RetentionPolicy.RUNTIME)
2public @interface Foo {}
3
4@FooContainer({@Foo, @Foo})
5public class Service {}
1@Repeatable(FooContainer.class)
2@Retention(RetentionPolicy.RUNTIME)
3public @interface Foo {}
4
5@Foo
6@Foo
7public class Service {}

It is guaranteed by the language specification that both syntaxes are equivalent, to the extent that the generated bytecode of Service class must be identical in both cases:

If a declaration context or type context has multiple annotations of a repeatable annotation type T, then it is as if the context has no explicitly declared annotations of type T and one implicitly declared annotation of the containing annotation type of T.

The realization is that the left syntax has long been how people implement @Repeatable before Java 8, and Java 8’s @Repeatable does nothing more than facilitate a long-existing practice.

Caveat: the equivalence only applies to multiple occurrences of a repeatable annotation; the following generates different bytecode:

1@FooContainer({@Foo})
2public class Service {}
3
4@Foo
5public class Service {}

AnnotatedElement

JDK uses interface AnnotatedElement (java.lang.reflect) to represent any construct that can be annotated. Well-known reflective representations of Java’s construct like java.lang.Class, java.lang.reflect.Method, java.lang.reflect.Field, etc. all implements this interface.

Many primitive APIs are exposed by this interface, useful for creating your annotation parser. So let’s first take a look into it:

for definition of repeatable-respecting API, see Repeatable Annotation Headache
