Using Apache Log4j2

Published at: December 30, 2023 | Last modification: April 25, 2024

java log4j2

The setup:

  1. Support different Log destination: console / file / mongo / sql / message queue / …
  2. Log format: raw text / CVS / MS xlsx / HTML / …
  3. Log file rolling by day / max file size.
  4. Configurable log line pattern: timestamp, line number, reporting class / method, etc.
  5. Reusable configuration unit.

Hierarchical LoggerConfig and “Additivity”

In log4j2, the one and only hierarchical relationship exists among org.apache.logging.log4j.core.config.LoggerConfig instances, established via the private LoggerConfig parent field.

Each LoggerConfig has a private final String name fields.

  • A LoggerConfig is said to be an ancestor of another iff. dot-separated segments of the prior’s name is a prefix of the latter.
  • A special LoggerConfig instance, known as the root LoggerConfig, is the ancestor of all LoggerConfigs, and always exists.

The Root LoggerConfig

Each interface Configuration, thus each LoggerContext, has one and only one root, stipulated by org.apache.logging.log4j.core.config.AbstractConfiguration#root.

It’s name is:

1public class LogManager {
2    public static final String ROOT_LOGGER_NAME = "";
3}

A Logger linked to the root LoggerConfig can be obtained by:

1// both equivalent
2Logger logger = LogManager.getLogger(LogManager.ROOT_LOGGER_NAME);
3Logger logger = LogManager.getRootLogger();

Deployer should configure root logger in the config file:

1<!-- ironically, LoggerConfig instances are configured under the <Loggers> tag -->
2<Configuration>
3    <Loggers>
4        <!-- <Root> must be after all <Logger>s -->
5        <Root>
6            <AppenderRef ref="MyConsoleAppender"></AppenderRef>
7        </Root>
8    </Loggers>
9</Configuration>

If root LoggerConfig is not configured, a default root LoggerConfig, which has a level of ERROR and has a Console appender attached, will be used.

Additivity

Additivity is a boolean attribute on each LoggerConfig, determining whether a logging event should bubble up to its parent (direct ancestor).

The metal model of event bubbling does have some similarity with configuration inheritance. It feels like a child LoggerConfig “inherit” appenders and other configs from an ancestor LoggerConfig, and additivity=false disables such inheritance.

Configurations

Discovery and priority of config files are:

  1. -Dlog4j2.configurationFile JVM property
  2. log4j2-test.<EXT> on classpath
  3. log4j2.<EXT> on classpath
  4. org.apache.logging.log4j.core.config.DefaultConfiguration, This will cause logging output to go to the console.

Possible <EXT>, ordered by priority: .properties,.[yaml|yml],.[json|jsn],.xml.

XML

2 flavors of XML config are supported: concise v.s. strict

 1    <Appenders>
 2        <!-- strict flavor -->
 3        <Appender type="Console" name="MyConsoleAppender">
 4            <Layout type="PatternLayout" pattern="%m MDC%X%n"/>
 5        </Appender>
 6        
 7        <!-- equivalent concise flavor -->
 8        <Console name="MyConsoleAppender" target="SYSTEM_OUT">
 9            <PatternLayout pattern="%m MDC%X%n"/>
10        </Console>
11    </Appenders>

Concise flavor use name of component class (which is extensible, i.e. new custom appender class can be authored) as tag name, thus it make less sense to provide xsd for concise flavor.

xsd for strict flavor:

 1<?xml version="1.0" encoding="UTF-8"?>
 2<Configuration
 3    strict="true"
 4    status="WARN"
 5
 6    xmlns="http://logging.apache.org/log4j/2.0/config"
 7    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
 8    xsi:schemaLocation="http://logging.apache.org/log4j/2.0/config
 9    https://raw.githubusercontent.com/apache/logging-log4j2/master/log4j-core/src/main/resources/Log4j-config.xsd">
10</Configuration>

When using strict flavor, <Configuration strict="true" ... > is required. Otherwise parser will fail at runtime.

Internal Designs

log4j2 is divided into maven artifacts:

  • org.apache.logging.log4j.log4j-api, the “facade”
  • org.apache.logging.log4j.log4j-core

对于简单的使用场景，compile时的 classpath 上只需要有 log4j-api 即可；但是也存在编译时就不得不依赖 log4j-core 的情况，如 https://logging.apache.org/log4j/2.x/manual/customconfig.html

运行时的 classpath 上需要有 log4j-core, 会以 SPI 的方式被加载。

Logger & LoggerConfig

Since log4j 2, each Logger instance merely hold a name and a reference to a LoggerConfig.

LoggerConfig is an activation of a Configuration.

All hierarchy / config inherntance happen among LoggerConfig instances.

A LoggerConfig is said to be an ancestor of another LoggerConfig if its name followed by a dot is a prefix of the descendant logger name. A LoggerConfig is said to be a parent of a child LoggerConfig if there are no ancestors between itself and the descendant LoggerConfig.

The one and only way to obtain a Logger is by LogManager.getLogger(name), which always return the same logger per name / class.

Each LoggerConfig has a org.apache.logging.log4j.core.config.LoggerConfig#level field, i.e. the threshold.

Log4j2 Plugin

A plugin is a class marked by @org.apache.logging.log4j.core.config.plugins.Plugin annotation.

Notable PatternLayout placeholders

  • %l, %location: complete caller class and line number, e.g. org.example.App.main(App.java:13)

    Generating location information is an expensive operation and may impact performance. Use with caution.

  • %d{pattern}, %date{pattern} time of logging

    e.g. %date{yyyy-MM-dd HH:mm:ss} -> 2022-03-29 15:23:50

  • %n: platform line separator

Class-specific Logger

Both log4j2 and slf4j recommends the following syntax to get a Logger instance:

1public class App {
2    private static final org.slf4j.Logger log = org.slf4j.LoggerFactory.getLogger(App.class);
3    private static final org.apache.logging.log4j.Logger log2 = org.apache.logging.log4j.LogManager.getLogger(App.class);
4}

In log4j2, this effectively use the FQCN of the argument class, and see if it could hit the name of an existing LoggerConfig:

1public interface LoggerContext {
2    default ExtendedLogger getLogger(Class<?> cls) {
3        final String canonicalName = cls.getCanonicalName();
4        return getLogger(canonicalName != null ? canonicalName : cls.getName());
5    }
6}

If the deployer does configure a LoggerConfig of this name, those rules will apply:

1    <Loggers>
2        <Logger name="org.example.App">
3            <AppenderRef ref="myConsoleAppender"/>
4        </Logger>
5    </Loggers>

Internally, this fetch the same org.apache.logging.log4j.Logger instance associated with this FQCN, or create a new if first called with this FQCN.
